MWAA Airflow orchestration path

The pipeline control plane for Silver and Gold.

This repository owns the Airflow DAG that coordinates six parallel Glue jobs, validates Silver row counts, refreshes the Glue Catalog, runs dbt against Athena, validates Gold, and leaves the platform ready for Streamlit, the custom HTML dashboard, and Slack analytics.

Full platform context

MWAA coordinates the Silver-to-Gold pipeline without owning the data transformations.

Control plane onlyMWAA is not the transformation engine. It calls managed services and decides when the next stage can run.
Clear data pathRaw data moves Bronze to Silver to Gold. The diagram keeps that path horizontal so the ownership is easy to follow.
Agent readinessGold tables and dbt artifacts become the downstream context used by Streamlit, HTML, and Slack through the analytics backend.
DAG graph

The Airflow graph fans out for Silver, then gates Gold sequentially.

Fan-outThe six Glue tasks run together because each reads and writes separate table prefixes.
Fan-inThe `silver_complete` join ensures no downstream work starts until every Silver table succeeded.
Sequential GoldCatalog refresh, freshness, dbt run, row-count validation, and dbt tests happen in a fixed order.
Validation gates

The DAG stops before producing misleading Gold data.

O1 protects SilverCloudWatch row-count metrics catch zero-row or missing-output failures before the Crawler and dbt spend time.
D2 protects freshnessdbt source tests verify that Silver reached the expected reference cutoff before Gold models run.
D3 protects joinsStaging counts from `run_results.json` are compared against Silver counts with a 5% tolerance.
Deployment ownership

DAG changes deploy in seconds; package changes stay in Terraform.

DAG-only deployThis repo’s workflow uploads DAG files to S3 and does not call `aws mwaa update-environment`.
Package changes are infraPython dependency changes are deliberate Terraform changes because they mutate the MWAA runtime environment.
dbt stays decoupleddbt project changes sync separately and are downloaded by the DAG when the Gold tasks run.
Orchestration options

MWAA and Step Functions run the same logical pipeline for different operating modes.

Step FunctionsStarts immediately and is the faster path for normal dev sessions where you want the platform up briefly.
MWAAGives the Airflow graph and managed scheduler experience, but its environment lifecycle is heavier.
Same contractBoth orchestrators enforce the same pipeline shape and feed the same analytics surfaces.

Repository mental model

Schedule → Parallel Silver → Gates → Gold

The repo is an ordering and evidence layer, not a transformation library.

Why max_active_runs is one

No overlap → No duplicate writes

Only one DAG run can write Silver and Gold at a time, which protects the data lake from overlapping runs.
